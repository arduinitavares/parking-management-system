# Bounded Context Definitions

This document translates the identified Business Subdomains into architectural Bounded Contexts for the EasyParkPlus Parking Management System. Each context establishes a unified Ubiquitous Language and formal boundary constraint for future microservice development.

## 1. Parking Allocation Context
- **Mapped Subdomain:** Core Domain (Parking Allocation & Resource Management)
- **Purpose Statement:** To act as the authoritative source of truth for the real-time occupation state of the parking facilities. This context calculates vehicle allocation validity, guarantees physical slot constraints (e.g., standard vs. EV), and manages the active check-in/check-out lifecycle of every vehicle transiting the system.

## 2. Facility Management Context
- **Mapped Subdomain:** Supporting Subdomain (Facility Hierarchy Management)
- **Purpose Statement:** To define and maintain the static physical topology of the parking network. This context manages the structural blueprint data—including facility addresses, floor layouts, capacities, and zone boundaries—serving as the foundational constraint map for the Parking Allocation Context.

## 3. EV Charging Context
- **Mapped Subdomain:** Supporting Subdomain (EV Charging Station Management)
- **Purpose Statement:** To orchestrate the telemetry and hardware lifecycles of electric vehicle chargers. This context tracks physical charger status (online/offline/faulted), publishes charging session progressions (e.g., battery levels), and operates independently of the core allocation routing rules.

## 4. Identity & Access Management (IAM) Context
- **Mapped Subdomain:** Generic Subdomain (Identity & Access Management)
- **Purpose Statement:** To secure administrative access to the overarching system. This context handles personnel authentication, secures domain boundaries via role-based access control, and issues the geographic or facility-scoped authorization tokens required to perform management actions.

## 5. Audit & Logging Context
- **Mapped Subdomain:** Generic Subdomain (Audit & Activity Logging)
- **Purpose Statement:** To provide an immutable, chronologically ordered ledger of system mutations. This context captures distributed domain events generated by other boundaries (like capacity overrides or hardware faults), enabling retroactive compliance reviews and technical troubleshooting.

## 6. Notification Context
- **Mapped Subdomain:** Generic Subdomain (Notification & Messaging)
- **Purpose Statement:** To decouple internal event broadcasting from local presentation layers. This context is responsible for routing real-time alerts (e.g., slot occupancy updates, charger failures) to the appropriate UI clients or facility attendants based on regional context.
